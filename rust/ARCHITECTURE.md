# GpgFrontend Rust Engine — Architecture

The `rust/` directory is a self-contained Rust crate (`gf-rust`) that provides
OpenPGP operations to the C++ core of GpgFrontend. It is compiled as a static
library (`libgf_rust.a`) and its public C API is generated automatically at
build time by **cbindgen** into `src/core/GpgCoreRust.h`.

---

## High-level picture

```
C++ core (GpgFrontend)
        │
        │  calls via GpgCoreRust.h (plain C functions)
        ▼
┌─────────────────────────────────────────────────┐
│                 ffi/  (FFI boundary)            │
│  ffi/mod.rs      – runtime (version, logger)    │
│  ffi/crypto/     – encrypt / decrypt / sign /   │
│                    verify entry points          │
│  ffi/key.rs      – key block manipulation       │
│  ffi/keygen.rs   – key generation               │
│  ffi/mem.rs      – memory deallocation          │
│  ffi/user_id.rs  – user ID operations           │
└───────────────────┬─────────────────────────────┘
                    │  calls internal Rust functions
        ┌───────────┴────────────────────┐
        │           Implementation       │
        │  crypto/   – all PGP ops       │
        │  key.rs    – key block ops     │
        │  keygen.rs – key generation    │
        │  user_id.rs– UID management    │
        │  utils.rs  – shared helpers    │
        │  cache.rs  – password cache    │
        │  err.rs    – error handling    │
        │  tar.rs    – archive helpers   │
        │  types.rs  – C-compat types    │
        └────────────────────────────────┘
```

The FFI layer is thin: it converts raw pointers to Rust types, calls an
internal function, converts the result back to raw pointers, and returns a
`GfrStatus` code. No business logic lives in `ffi/`.

## Module reference

### `types.rs`

All `#[repr(C)]` structs and enums that cross the FFI boundary. The C++ side
mirrors every type declared here in `GpgCoreRust.h` (generated by cbindgen).
Key items:

| Type                  | Purpose                                                                 |
| --------------------- | ----------------------------------------------------------------------- |
| `GfrStatus`           | Return code for every exported function (0 = success, negative = error) |
| `GfrKeyAlgo`          | Public-key algorithm selector used for key generation and metadata      |
| `GfrSignMode`         | Inline / ClearText / Detached signing modes                             |
| `GfrRevocationCode`   | Reason codes for revocation signatures                                  |
| `GfrKeyConfig`        | Primary key or subkey generation parameters                             |
| `GfrKeyMetadataC`     | Full key metadata returned to C++                                       |
| `GfrXxxResultC`       | Structs holding the output of each crypto operation                     |
| `GfrPublicKeyFetchCb` | Callback: C++ provides armored public key by fingerprint                |
| `GfrSecretKeyFetchCb` | Callback: C++ provides armored secret key by key ID                     |
| `GfrPasswordFetchCb`  | Callback: C++ prompts user for a passphrase                             |
| `GfrFreeCb`           | Callback: C++ frees memory it allocated in response to a fetch callback |

### `ffi/` — FFI boundary

Every function here is `pub extern "C"` with `#[unsafe(no_mangle)]`. The
pattern used by every entry point is:

```rust
pub extern "C" fn gfr_crypto_xxx(...) -> GfrStatus {
    let result = catch_unwind(|| -> Result<(), GfrStatus> {
        // 1. null-pointer guard
        // 2. convert C types → Rust types
        // 3. call internal function
        // 4. convert result → C types (leak ownership to C++)
        Ok(())
    });
    match result {
        Ok(Ok(_))  => GfrStatus::Success,
        Ok(Err(e)) => e,
        Err(_)     => GfrStatus::ErrorPanic,   // Rust panic caught here
    }
}
```

Key sub-modules:

- **`ffi/crypto/encrypt.rs`** — `gfr_crypto_encrypt_*`, `gfr_crypto_encrypt_and_sign_*`, symmetric variants
- **`ffi/crypto/decrypt.rs`** — `gfr_crypto_decrypt_*`, `gfr_crypto_decrypt_and_verify_*`
- **`ffi/crypto/sign.rs`** — `gfr_crypto_sign_*`, `gfr_crypto_verify_*`, `gfr_crypto_get_signature_issuers`
- **`ffi/key.rs`** — metadata extraction, public-key derivation, recipient sniffing, password change, subkey add/delete/revoke, key merge, revocation certificate generation/import
- **`ffi/keygen.rs`** — `gfr_crypto_generate_key`, `gfr_crypto_add_subkey`
- **`ffi/mem.rs`** — one `gfr_crypto_free_*` function per result struct; C++ **must** use the matching free function for every pointer the engine returns
- **`ffi/user_id.rs`** — add, delete, update, set-primary, revoke user IDs
- **`ffi/mod.rs`** — `gfr_rust_hello`, `gfr_rust_engine_version`, `gfr_init_logger`

### `crypto/` — OpenPGP operations

The unified crypto module. `mod.rs` acts as a shared header: it re-exports
all common types and imports so that each sub-module can write `use super::*;`
and get everything it needs.

- **`mod.rs`** — shared result types, common imports, and helper functions
  (`with_signing_key`, `parse_signer_block`, `sniff_*`, `cert_contains_issuer`, …)
- **`encrypt.rs`** — public-key encrypt, encrypt-and-sign, symmetric encrypt; streaming + in-memory
- **`decrypt.rs`** — decrypt-and-verify, archive decrypt; streaming + in-memory
- **`sign.rs`** — Inline / ClearText / Detached signing; streaming + in-memory
- **`verify.rs`** — detached-signature verification over seekable streams; in-memory verify

Each operation family has both a **streaming** variant (takes `R: Read`, `W: Write`)
and an **in-memory** variant (takes `&[u8]`, returns `Vec<u8>`). The in-memory
variants are thin wrappers that create a `Cursor` and delegate to streaming.

### `key.rs`

Pure key-block manipulation on armored strings — no crypto operations.
Covers: metadata extraction (`extract_metadata_many_internal`), public-key
derivation, block merging, passphrase change, subkey deletion/revocation,
revocation certificate generation and import.

The merge logic (`merge_key_block_internal`) handles all four combinations of
(secret | public) × (secret | public) input, resolving to the richest output.

### `keygen.rs`

Key generation. `keygen_dynamic` builds a `SignedSecretKey` from a
`GfrKeyConfig`; `create_key_internal` wraps it with optional passphrase
protection and armored export; `add_subkey_internal` attaches a new subkey to
an existing key.

Post-quantum hybrid algorithms (ML-KEM, ML-DSA, SLH-DSA) require OpenPGP v6
keys — the v6 flag is forced automatically whenever any key in the request uses
a PQC algorithm.

### `user_id.rs`

User ID management: add, delete, update (rename), set-primary, revoke. All
operations require a secret key block (signing is needed). `delete_user_id`
also accepts public-only blocks.

### `utils.rs`

Shared helpers used across `crypto`, `key`, `keygen`, and `user_id`:

- **`resolve_key_type`** — maps `GfrKeyAlgo` + `can_encrypt` to an rPGP `KeyType`. The `can_encrypt` flag is what distinguishes Ed25519 from ECDH Curve25519 — both use the same 25519 parameter set.
- **`determine_algo`** / **`extract_key_length`** — reverse-map rPGP `PublicParams` back to `GfrKeyAlgo`.
- **`fetch_password_internal`** / **`fetch_password_with_cache`** — invoke the C++ passphrase callback and optionally cache the result.
- **`choose_template_self_sig`** — pick the best existing self-signature as a template when re-signing a user ID (prefers the newest sig that carries a `KeyFlags` subpacket).
- **`build_revocation_reason_subpacket`** — construct the revocation reason subpacket from a `GfrRevocationCode`.

### `cache.rs`

Thread-safe, TTL-based passphrase cache. The global `PASSWORD_CACHE` instance
has a 5-minute TTL. Cache entries are keyed by `(channel, fingerprint, info)`
and are zeroed on eviction or drop.

Three policies control cache access per operation:

- `Default` — read cache first; on miss, fetch and store.
- `Bypass` — skip entirely. Used for symmetric operations (no fingerprint to key on) and for "set new password" flows.
- `Refresh` — evict, re-prompt, store. Used after a wrong-passphrase failure.

### `err.rs`

Thread-local `LAST_ERROR` slot. Every internal function that returns a
`GfrStatus` error also calls `set_last_error` so that C++ can retrieve the
human-readable message via `gfr_get_last_error_msg`.

The `IntoGfrResult` trait maps rPGP and `std::io` errors to `GfrStatus` codes,
logging them as a side effect.

### `tar.rs`

Single function: `build_tar_tempfile_from_directory` packs a directory into an
anonymous temporary file (auto-deleted on drop) and returns the file handle
seeked to offset 0, ready for the caller to read.

## FFI contract

### Memory ownership

The engine heap-allocates all output data (strings, arrays, structs with
pointer fields) and **transfers ownership to the C++ caller**. The caller
**must** free every pointer using the corresponding `gfr_crypto_free_*`
function from `ffi/mem.rs`. Passing pointers to `free()` or any other
allocator is undefined behaviour.

```
gfr_crypto_free_string          – for *mut c_char from any gfr_* function
gfr_crypto_free_buffer          – for raw byte buffers (data/data_len pairs)
gfr_crypto_free_encrypt_result  – for GfrEncryptResultC
gfr_crypto_free_decrypt_result  – for GfrDecryptResultC
gfr_crypto_free_sign_result     – for GfrSignResultC
gfr_crypto_free_verify_result   – for GfrVerifyResultC
gfr_crypto_free_encrypt_and_sign_result
gfr_crypto_free_decrypt_and_verify_result
gfr_crypto_free_key_generate_result
gfr_free_metadata_array         – for GfrKeyMetadataC arrays
```

### Callbacks

Four function-pointer types let C++ provide data to the engine at call time:

| Callback              | Direction    | Purpose                                                |
| --------------------- | ------------ | ------------------------------------------------------ |
| `GfrPublicKeyFetchCb` | C++ → Engine | Return armored public key block by fingerprint         |
| `GfrSecretKeyFetchCb` | C++ → Engine | Return armored secret key block by key ID              |
| `GfrPasswordFetchCb`  | C++ → Engine | Prompt user; write passphrase bytes; return byte count |
| `GfrFreeCb`           | Engine → C++ | Free memory allocated by a fetch callback              |

The engine frees callback-allocated memory by calling `GfrFreeCb` immediately
after copying the data into Rust-owned storage.

### Error handling

Every exported function returns `GfrStatus`. On error, the engine also writes a
detailed message to the thread-local last-error slot. C++ should call
`gfr_get_last_error_msg()` (and free the result with `gfr_crypto_free_string`)
whenever a function returns a non-zero status.

## Key design decisions

### Signing key selection — the `!` protocol

When passing a secret key block to a signing function, the caller may prepend a
fingerprint (or key ID) followed by `!` to force use of a specific subkey:

```
"AABBCCDD112233445566!\n-----BEGIN PGP PRIVATE KEY BLOCK-----\n..."
```

This is parsed by `parse_signer_block` in `crypto/mod.rs`. Without the prefix,
`with_signing_key` selects the first capable signing subkey automatically.

### Inline mode uses all signers; ClearText/Detached use only the first

In `crypto/sign.rs`, the Inline mode iterates all provided secret key blocks
and adds each as a signer to the streaming builder in one pass. ClearText and
Detached modes buffer the full input to memory (required by rPGP's API) and
only use the first key — the output format does not compose cleanly with
multiple independent signers in a single-pass stream.

### Anonymous recipients

A recipient key ID of `"0000000000000000"` means the message was encrypted
without embedding recipient IDs (hidden recipients). The engine accepts this
and attempts decryption with any available secret key.

### Password cache bypass for symmetric operations

Symmetric encryption/decryption uses `PasswordCachePolicy::Bypass` because
there is no fingerprint to key the cache on. The cache entry would create false
hits across unrelated symmetric messages.

## Build notes

### cbindgen generates the C header

`build.rs` runs cbindgen over the entire source tree and writes
`src/core/GpgCoreRust.h`. cbindgen reads every `.rs` file — not just the
public ones — so syntax that cbindgen's parser does not support will break the
build even in private modules.

**Known cbindgen constraint (v0.29.2):** Do **not** use `if let … && let …`
(let-chain syntax, stabilised in Rust 1.87). Use nested `if let` instead.
Violating this causes `ParseSyntaxError` at build time.

### Crate-level lint allowances

Three lints are suppressed in `lib.rs` for legitimate reasons:

- **`not_unsafe_ptr_arg_deref`** — FFI functions null-check before dereferencing; marking
  40+ `extern "C"` functions `unsafe` would break the C caller.
- **`too_many_arguments`** — FFI functions mirror C entry points 1-to-1; grouping args would change the ABI.
- **`collapsible_if`** — nested `if let` is kept where let-chains cannot be used (cbindgen constraint).

## Adding a new operation

1. **Add types** to `types.rs` if the operation needs new C-visible structs.
2. **Implement the logic** in the appropriate `crypto/` sub-module (`encrypt.rs`, `decrypt.rs`, `sign.rs`, or `verify.rs`). Follow the existing pattern: streaming function first, then a thin in-memory wrapper that wraps the data in a `Cursor`.
3. **Add the FFI entry point** to the appropriate `ffi/crypto/` file. Follow the `catch_unwind` + null-check + delegate + leak pattern.
4. **Add a free function** in `ffi/mem.rs` if the new function returns heap-allocated data.
5. **Run `cargo build`** — cbindgen regenerates the header automatically. Verify the new symbol appears in `src/core/GpgCoreRust.h`.
6. **Run `cargo clippy`** — the baseline is zero warnings.
